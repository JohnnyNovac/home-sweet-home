# Код-ревью home-sweet-home

Цель — найти дефекты до того, как они застынут под новыми слоями. Это ревью существующего кода и
изменений, с упором на то, что в обычном diff-ревью теряется: поведение системы в долгом прогоне с
реальным железом.

Ты НЕ правишь код в этом скилле. Ты собираешь приоритизированный список находок со ссылками
`file:line`. Правки — отдельным проходом, по явному запросу пользователя.

## 1. Определи объём

- Если передан аргумент — это область ревью:
    - имя модуля (`event-service`, `presence-service`, `yandex-service`, `shared`, `grpc-api`) — ревьюй весь модуль;
    - путь к файлу/каталогу — ревьюй его;
    - `all` — пройди модули в порядке риска: `event-service` → `presence-service` → `yandex-service` → `shared` →
      `grpc-api`.
- Без аргумента — ревьюй незакоммиченные изменения (`git status`, `git diff` + `git diff --staged`).
  Если рабочее дерево чистое — ревьюй последний коммит (`git show HEAD`).

Сначала собери список затронутых файлов, потом прочитай их и достаточный контекст вокруг (вызывающий
код, конфиг, тесты). Не ревьюй по одному diff-хунку без контекста.

## 2. Чек-лист

Прогоняй каждый пункт. Пункты про долгоживучесть и конкурентность — главная ценность этого скилла,
не пропускай их даже если изменение выглядит локальным.

**Корректность**

- Граничные случаи парсинга (`JsonDtoParser`, `MeasurementDTO`): пустой/битый JSON, отсутствующие поля, неожиданный тип.
- Парсинг `deviceId`/`sensorType` из routing key (`parts[1]` и т.п.) — что будет при неожиданном формате ключа.
- Логика диспетчеризации в `EventRunner` / `SensorHandlerFactory` — неизвестный `sensorType`, отсутствующий хендлер.

**Долгоживучесть (IoT, долгий прогон)**

- Переподключения MQTT и AMQP: восстанавливается ли подписка/листенер после обрыва, не теряются ли сообщения.
- Освобождение ресурсов: закрытие `MqttClient`, gRPC-каналов, реактивных подписок; нет ли утечек на каждом сообщении.
- Неограниченный рост в памяти: `knownDeviceIds` и любые кэши/Map, которые только пополняются.
- Идемпотентность при переполучении сообщения (AMQP redelivery) — не ломается ли состояние/HA-discovery при дубле.
- Поведение при рестарте Home Assistant и брокера: гейт по `app.ha.status-topic`, повторная публикация discovery при
  `online`.

**Конкурентность**

- Разделяемое изменяемое состояние (`knownDeviceIds`, поля хендлеров, registry) при параллельной обработке сообщений
  листенерами.
- Атомарность upsert-ов в `DeviceRegistry` (`recordSeen`, ленивое заполнение `sensorType`).

**Реактивный Mongo (WebFlux)**

- Блокирующие вызовы внутри реактивных цепочек (`.block()`, синхронный I/O).
- Неподписанные `Mono`/`Flux` (цепочка собрана, но не выполняется).
- Обработка ошибок в цепочке — проглоченный `onError`, отсутствие fallback.

**Обработка ошибок**

- Проглоченные исключения (пустой `catch`, `catch` без логирования контекста).
- Поведение при недоступности downstream: gRPC-вызов `yandex-service`, Yandex HTTP API — таймауты, ретраи, что видит
  вызывающий.
- ack/nack логика листенеров: что происходит с сообщением при исключении в обработчике.

**Проектные конвенции (из CLAUDE.md)**

- Новое свойство должно идти в соответствующий `@ConfigurationProperties`-класс, а не через `@Value`.
- Jackson 3 (`tools.jackson.*`), не `com.fasterxml.jackson.*`.
- `deviceId` только в routing key, никогда в JSON-теле.
- Имена очередей/топиков вынесены в `application.yml`, не захардкожены.
- Тесты: не ломать java-agent Mockito и гейты `app.mqtt.enabled` / исключение RabbitAutoConfiguration в тестовом
  конфиге.

## 3. Отчёт

Выдай находки, сгруппированные по серьёзности. Для каждой:

- `severity` — Critical / High / Medium / Low.
- `file:line` — кликабельная ссылка на место.
- Что не так и при каких условиях проявится (особенно: «только в долгом прогоне», «только при обрыве связи»).
- Конкретное предложение по исправлению (одной-двумя фразами, без написания кода).

Если находок нет в какой-то категории — так и скажи, не выдумывай. В конце — короткое резюме: что
выглядит наиболее рискованным и с чего начать правки.
